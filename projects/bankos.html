Challenges
& Solutions

Thread Safety

Managing lock allocation across multiple account nodes risks severe deadlock bugs where threads wait on circular resources indefinitely.

Challenge: Circular Deadlocks during Simultaneous Mutual Transfers

If Thread 1 locks Account A and tries to lock B (for A -> B), while Thread 2 locks Account B and tries to lock A (for B -> A), the threads enter a circular dependency, freezing the transaction core permanently.

Fix: Strict Lock-Ordering & Banker's Algorithm Allocation

We implemented a strict lock acquisition rule: locks are always acquired in ascending order of Account IDs (e.g. if a thread needs to transfer between A and B, it must always lock the smaller ID first, regardless of transaction direction). Additionally, we modeled a Banker's Algorithm allocation validator to analyze multi-account lock demands in parallel, ensuring circular dependencies can never form in the execution pool.
